5. RELATIONSHIPS SUMMARY
=========================

Entity ──────────► EntityVisitor (accepts)
   │
   │ inheritance
   │
   ├──► Monster
   ├──► Item
   └──► Clue

EntityVisitor ◄──── AttackAction   (implements)
              ◄──── CollectAction  (implements)
              ◄──── ExamineAction  (implements)

AttackAction   ──► Player (uses)
CollectAction  ──► Player (uses)
ExamineAction  ──► Player (uses)

Dungeon ◄────── Room (contains 0..*)
Room ◄────── Room (connects to 0..*, tree structure)
Room ◄────── Entity (contains 0..*)


6. DESIGN PATTERN NOTES
========================

VISITOR PATTERN:
- Allows different actions to be performed on entities
- Actions: Attack, Collect, Examine
- Entities accept visitors via double dispatch
- New actions can be added without modifying Entity classes
- Demonstrates separation of concerns

TREE STRUCTURE:
- Rooms are nodes connected by edges (doors)
- Dungeon maintains reference to root (entrance)
- Parent-child relationships represent room connections
- No cycles (proper tree structure)
- Supports depth-first traversal via player navigation
